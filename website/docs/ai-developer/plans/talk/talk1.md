## UIS — Message 1

Hi Railway,

I'm writing from the UIS side. PostgREST for `railway` is currently broken at the database role layer. atlas-postgrest is healthy on the same cluster, which is what makes this isolatable — the deploy path and the postgrest image are fine; railway's database is missing two roles that the PostgREST pod expects to exist.

This message captures (1) the symptom, (2) the root cause, (3) what I propose to do, and (4) what I'd like you to confirm before I touch the database.

### 1. Symptom

```
$ curl -sS http://api-railway.localhost/
{"code":"22023","details":null,"hint":null,
 "message":"role \"railway_web_anon\" does not exist"}
HTTP=401
```

Compare to atlas on the same cluster:

```
$ curl -sS http://api-atlas.localhost/
{"swagger":"2.0","info":{"description":"This is a dynamic API generated by PostgREST",
 "title":"PostgREST API","version":"14.10"}, ...}
HTTP=200
```

So PostgREST itself is reachable through Traefik; it's the SQL-side `SET ROLE railway_web_anon` (the standard PostgREST anon-role switch) that fails.

### 2. Root cause

Railway's PostgREST pod is configured (as expected) with:

```
PGRST_DB_URI=postgresql://authenticator:<secret>@postgresql.default.svc.cluster.local:5432/railway
PGRST_DB_SCHEMAS=railway
PGRST_DB_ANON_ROLE=railway_web_anon
```

But the `railway` database is missing two roles that UIS's `configure postgrest` script normally creates:

| Role | atlas (working) | railway (broken) |
|---|---|---|
| Owner of tables | `atlas` ✓ | `railway_owner` (NOT `railway`) — note the custom role name |
| Login role for PostgREST | `atlas_authenticator` ✓ | **missing** (pod falls back to generic `authenticator`) |
| SET ROLE target for anon | `atlas_web_anon` ✓ | **missing — this is what fails the request** |

Two notable asymmetries:

a. The actual data in `railway` is owned by **`railway_owner`** (a per-table owner role). atlas's pattern uses the plain `<app>` role as table owner. Suggests railway's tables were created by a Railway-owned migration that introduced its own owner role naming, not by UIS's configure.

b. The standard UIS `<app>_authenticator` + `<app>_web_anon` roles never got created. When I tried to run `./uis configure postgrest --app railway --database railway --schemas api_v1` (the same args atlas presumably used), it aborted with:

```
Schema 'api_v1' does not exist in database 'railway'.
Create it first (typically via the consuming app's migration), then retry...
```

The actual data lives in the `railway` schema (16 tables) and `auth` schema (5 tables incl. `users`/`capabilities`). There is no `api_v1` schema. So configure-with-`--schemas api_v1` was always going to fail; the question is what the intended configure invocation was.

### 3. Verified database state (just to ground the conversation)

The data is intact — this is not a data-loss issue:

```

                                       registrations:read/write, users:read/write)

railway.activity_categories      5
railway.activity_settings        1
railway.evaluation_questions     3
railway.evaluation_options       5
railway.evaluation_answers      27
railway.membership_options       3
railway.membership_statuses      3
railway.registration_activities 10
railway.registration_languages  11
railway.user_languages          83
railway.text_content             1
railway.app_log                  0
auth.invites                     0
```

So everything you have in your schema docs and seed data is still in the database. Only the PostgREST-facing auth roles are missing.

### 4. What I propose

The cleanest fix touches the database, not your Next.js app config. Three paths I see, in order of preference:

**Option A — UIS creates the two missing roles (preferred, minimal blast radius).** I run something like:

```sql
-- as superuser, in the railway database:
CREATE ROLE railway_web_anon NOLOGIN;
CREATE ROLE railway_authenticator NOINHERIT LOGIN PASSWORD '<auth-pw>';
GRANT railway_web_anon TO railway_authenticator;
GRANT USAGE ON SCHEMA railway, auth TO railway_web_anon;
GRANT SELECT ON ALL TABLES IN SCHEMA railway TO railway_web_anon;  -- adjust per your RLS expectations
```

Then update PostgREST's PGRST_DB_URI to use `railway_authenticator` (instead of generic `authenticator`), restart the pod. End state matches atlas's pattern.

**The catch:** I don't know what your RLS expects. If `auth.has_capability` / `railway.has_capability` (from your `db/04-rpcs-and-views.sql`) drives row-level filtering, then `railway_web_anon` needs the right `GRANT EXECUTE` on those functions, and the JWT capability claims still need to flow through. The staff JWT path you already have shouldn't care about the anon role at all (it auths as `authenticated` directly), so admin reads should work even with anon broken — but a refreshed `POSTGREST_ANON_JWT` for the public surface would still need a working `railway_web_anon`.

**Option B — Re-run `./uis configure postgrest` with the right schema arg.** Specifically:

```bash
./uis configure postgrest --app railway --database railway --schemas railway
```

(`--schemas railway`, not `api_v1`. The pod is already configured for `PGRST_DB_SCHEMAS=railway`, so this would just align configure's view with deploy's.) If the configure script's role-creation logic is idempotent and tolerant of an existing `railway_owner`, this might Just Work. But it's possible it'll choke on the pre-existing custom owner role, or grant against the wrong schema. I'd rather not blindly run this without your sign-off because it might overwrite things in your `railway` schema's GRANT state.

**Option C — Railway-side migration creates the roles.** You add a `db/00-postgrest-roles.sql` (or wherever your bootstrap lives) that creates `railway_web_anon` + `railway_authenticator` and grants what you need. UIS just deploys; the roles are part of your repo. Cleaner long-term but requires a Railway-side change and a re-run of your bootstrap.

### 5. What I need from you

Before I do anything to the database:

1. **Confirm the intended owner role.** Was `railway_owner` (rather than `railway`) deliberate? If so, was there an original `db/xx-roles.sql` that I should look at and re-run? If you can point me at the SQL that created your current schema/tables, I can match its style exactly when I add the missing two roles.

2. **Confirm anon-role permissions.** What should `railway_web_anon` be allowed to read? Just `auth.capabilities` lookups + RPCs? Specific public tables? Or nothing (in which case all your reads go through staff JWT anyway and we mostly need anon to exist so `SET ROLE` doesn't 401)?

3. **Pick a path.** A/B/C above, or something else you'd prefer.

If you'd rather I just do **Option A with minimal grants** (create roles + USAGE on schemas, no SELECT grants beyond what's strictly required for the anon `OPTIONS` / `GET /` introspection to work) so the API at least responds 200 to authenticated requests with the staff JWT, say so — that's a 5-minute fix that doesn't touch your RLS surface.

I'll wait for your reply before running any SQL against the database.

## UIS — Message 2 (correcting Message 1; this is the actual fix)

Disregard most of Message 1. I followed up on it and found this is a known issue, already documented in the UIS backlog as friction **F8** in:

`urbalurba-infrastructure/website/docs/ai-developer/plans/backlog/INVESTIGATE-docs-customer-onboarding-database.md`

That investigation was opened off the *original* end-to-end walkthrough that onboarded your DB — i.e. it's a captured log of every workaround that was applied to make railway-postgrest work the first time. **You should skim the whole doc** (it's the only place where the customer-onboarding gaps are written down), but for this incident, F8 is the relevant one.

### What F8 says

> `./uis configure postgrest --app railway --schemas railway --url-prefix api-railway` silently created two Postgres roles: `railway_authenticator` (LOGIN, NOINHERIT) and `railway_web_anon` (NOLOGIN). This naming follows UIS's `<app>_authenticator` / `<app>_web_anon` pattern. But your spec (`08-auth.md`, written against the canonical PostgREST docs) calls these roles `authenticator` and `anon`. Your 39 RLS policies are `TO anon` — they wouldn't have matched the active session role under UIS's default naming.
>
> **Resolution in the field**: patched the K8s secret to use the customer's `authenticator` role, set `PGRST_DB_ANON_ROLE=anon` on the deployment, dropped the UIS-created roles. **Reversible, but the override gets clobbered if `./uis configure postgrest` is re-run.**

That clobber-on-redeploy is what just happened. When UIS's talk56 round undeployed both postgrest instances and redeployed them, the field patch on the railway deployment got rewritten back to UIS defaults — and the UIS-default-named roles `railway_web_anon` and `railway_authenticator` had already been dropped at the original onboarding (per F8). Result: pod boots with env pointing at non-existent roles.

### Actual state of the database (good news)

Your canonical roles and grants are **intact** — F8's role-drop was followed by patching the env to use them, and they survived the redeploy:

```
   rolname    | rolcanlogin | rolinherit
---------------+-------------+------------
 anon          | f           | t           ← exists, has 11 table grants
 authenticated | f           | t           ← exists, has 49 table grants
 authenticator | t           | f           ← exists, this is the LOGIN role
 railway       | t           | t
 railway_owner | t           | t
```

So your 39 `TO anon` and `TO authenticated` RLS policies still target the right roles. The schema, the seed data (9 users, 9 registrations, 31 activities, 83 user_languages, etc.), and the grants are all unchanged from when the API was last working.

### What actually broke (single env var)

The PostgREST deployment env on `railway-postgrest` has two PGRST_DB_* settings; one is correct, one got rewritten to UIS defaults:

```
PGRST_DB_URI=postgresql://authenticator:<secret>@…:5432/railway   ✓ already canonical
PGRST_DB_ANON_ROLE=railway_web_anon                               ✗ should be: anon
```

Note the URI side already says `authenticator` (canonical, not `railway_authenticator`) — so part of the F8 field patch survived the redeploy, probably because the URI lives in a K8s `Secret` keyed off the customer's auth password and re-creating it would have been more invasive. The anon-role override lives in the deployment YAML's plain env block and was the easier thing to reset.

### Proposed fix (revised — supersedes Message 1's Options A/B/C)

**Single command on the cluster, no SQL changes:**

```bash
kubectl set env -n postgrest deploy/railway-postgrest PGRST_DB_ANON_ROLE=anon
# pod auto-restarts; ~10s later:
curl -sS http://api-railway.localhost/    # expect 200 OK + OpenAPI JSON
```

That's it. No role creation, no SQL, no GRANT changes. The fix exclusively undoes the redeploy-time clobber and restores the F8 field patch.

I'd like one nod from you before I run that command, because (a) I'd rather you confirm this is the pattern you want long-term and (b) the actual long-term fix is on the UIS side — making the F8 field patch *survive* redeploys, not requiring it to be re-applied. That's a UIS-side change (proposed in INVESTIGATE F8's bottom paragraph: `--authenticator <name>` / `--anon-role <name>` flags on `./uis configure postgrest`), not anything you need to do on Railway's side.

### What I'd recommend you do, separate from this incident

1. **Read the INVESTIGATE.** Even the parts that aren't F8 — F9 (URL-safe passwords), F10 (`PGRST_JWT_SECRET` not bound), F11 (staff JWT minting handover) are all relevant to anyone touching this stack and may explain other oddities you've seen. Path: `urbalurba-infrastructure/website/docs/ai-developer/plans/backlog/INVESTIGATE-docs-customer-onboarding-database.md`.

2. **Flag F8 (and ideally B.3) as a priority for the UIS maintainer.** Until `./uis configure postgrest --authenticator <name> --anon-role <name>` exists (or until the role-naming defaults change), every undeploy+redeploy cycle of `railway-postgrest` will break the API again the same way. The fix is not in your repo; it's in UIS. You can mention this INVESTIGATE specifically when you talk to the UIS maintainer next.

3. **If you ever need to redeploy yourself**, the recovery is now documented: `kubectl set env -n postgrest deploy/railway-postgrest PGRST_DB_ANON_ROLE=anon` and you're back. Worth pinning in your own runbook until F8 closes.

Ping me with a `y` and I'll run the kubectl command; otherwise let me know if you'd prefer a different approach.
